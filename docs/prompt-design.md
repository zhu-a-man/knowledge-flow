# AI 提取 Prompt 设计原理

本文记录 `ai_processor.py` 中提取 Prompt 的设计思路，解释「为什么这样写」而不是「写了什么」。

---

## 核心目标：让知识"日后找得回来"

Prompt 设计的出发点不是"提炼准确"，而是**"一个月后的自己能快速找到并看懂"**。

这两个目标的差异很大：
- "提炼准确"→ 倾向于完整、客观、术语准确
- "日后找得回来"→ 倾向于口语化、有钩子感、分类符合检索习惯

---

## 一、分类体系演进

### v1：自由分类（已弃用）

让 AI 自行发明 topic/dimension，完全无约束。

**问题**：
- 同义 topic 不收敛（"AI工具" / "AI应用" / "人工智能工具"三个节点说的是同一件事）
- 边界模糊（Cursor 既被归入"效率工具"又被归入"AI工具"）
- 粒度不一致（"编程"和"Python异步编程优化"同级出现）
- 知识越积累越散，树越长越乱

### v2：受控词表 + 分面分类法（当前版本）

基于三个经典理论重新设计：

**① MECE 原则**（McKinsey, 1960s）
- Mutually Exclusive：每条知识只归入一个领域，领域之间不重叠
- Collectively Exhaustive：所有知识都能被覆盖，不留死角
- 实践：8 个领域互斥穷尽，加"其他"兜底

**② 分面分类法**（Ranganathan, 1933; CRG, 1955）
- 不用单一维度硬分，而是用多个正交维度同时描述一条知识
- 领域（关于什么）× 子领域（具体方向）× 内容形式（什么类型）
- 用户可以从任意维度切入检索

**③ 受控词表 + 渐进扩展**（TnT-LLM, 2025）
- 给模型一张固定的类目表，要求从中选择
- 无法匹配时走"其他"兜底，人工确认后可扩展词表
- 既保证一致性（同类内容必然聚合），又保留扩展性

### 分类体系结构

```
领域（topic）      8 个 MECE 大类 + "其他"兜底
  └─ 子领域（dimension）  每个领域 4-5 个
内容形式（content_form）  6 种，跨领域通用
```

三个维度**正交独立**：
- 领域回答"关于什么"
- 子领域回答"具体哪个方向"
- 内容形式回答"是什么类型的内容"

词表定义在 `taxonomy.py` 中，修改类目只需编辑该文件。

## 二、消歧规则：核心价值驱动力

同一篇文章可能涉及多个领域，分类依据是**核心价值驱动力**而非表面涉及的领域。

| 内容 | 分类 | 原因 |
|------|------|------|
| 用 Cursor 写代码的技巧 | AI与大模型 > AI编程 | 核心价值是 AI 能力 |
| Python 语法入门教程 | 软件开发 > 编程语言与框架 | 核心价值是语言本身 |
| 用 AI 提升团队效率 | AI与大模型 > AI工具应用 | 核心驱动是 AI |
| GTD 时间管理法 | 效率与工具 > 工作方法论 | 核心是方法论本身 |

Prompt 中用 `boundary` 字段辅助消歧——每个领域附带一句边界说明。

## 三、为什么角色是"朋友"而不是"知识管理专家"

测试了大量 Prompt 后发现，角色设定对输出风格影响最大。

**"知识管理专家"角色产出**（书面腔，没有阅读欲望）：
```
该工具具备AI辅助代码补全功能，可显著提升开发效率，
适用于专业软件开发场景，支持多种编程语言
```

**"最懂你的朋友"角色产出**（口语化，读完有共鸣）：
```
Cursor — 写代码时 AI 实时帮你补全和改 bug，速度快到不像话
```

**原因**：大模型的风格跟随角色设定。"专家"激活的是学术/报告写作模式，"朋友"激活的是推荐/分享模式。知识库是给自己用的，后者更有效。

## 四、为什么用"好/差示例对比"而不是只说规则

单说规则（如"语言通俗易懂"），模型的遵从率低，因为"通俗"本身就是主观的。

加了对比示例后，模型有了锚点：

```
好示例：「Cursor — 写代码时 AI 实时补全和改 bug，速度快到不像话」
差示例：「Cursor — 一款具备AI辅助功能的代码编辑器」
```

**关键原则**：
- 好示例要体现"帮你解决什么问题"或"为什么值得一试"
- 差示例要是模型最容易产出的那种（功能描述型）
- 每种内容形式都给一对示例

## 五、为什么按内容形式区分提炼策略

不同形式的内容，提炼的逻辑完全不同：

| 形式 | 提炼重点 | 格式 |
|------|---------|------|
| 工具清单 | 原文有几条就提几条，一条不能少 | 名称 — 价值说明 |
| 观点洞察 / 方法论 | 3-5 个可复用的结论 | 行动建议或认知翻转句 |
| 教程步骤 | 关键步骤，"你去做"的语气 | 操作步骤 |
| 行业动态 | 发生了什么 + 为什么关注 | 事件 + 影响 |

没有分类时，模型会用统一格式处理所有内容，导致清单被压缩（原文10条只提5条）、观点被罗列而不是提炼。

---

## 六、清单类为什么要"原文有几条就提几条"

这是版本迭代中发现的最重要规则。

早期 Prompt 没有这条约束，用户反馈"公众号里讲了6个技能，只提取了5个"。

原因：模型默认会"精简"，觉得重复或价值低的条目可以省略。但对用户来说，**清单的完整性比精练更重要**——你不知道哪条会是他下次搜索的关键词。

---

## 七、summary 为什么要口语化

summary 显示在龙虾的回复里，是用户判断"值不值得点进去看知识库"的第一眼。

**书面腔 summary**：
```
本文介绍了多种提升工作效率的AI工具及其应用场景
```

**口语化 summary**：
```
几个工具装上去效率翻倍，收藏这篇
```

后者更像朋友推荐，有行动感，用户更可能点进去。

## 八、JSON 安全解析的必要性

即使 System Prompt 明确写"只返回 JSON，不要任何其他文字"，模型偶尔仍会返回：

````
```json
{"topic": "AI工具", ...}
```
````

直接 `json.loads()` 会抛异常。必须用正则预处理：

```python
def _parse_json_safely(raw: str) -> dict:
    raw = raw.strip()
    match = re.search(r"```(?:json)?\s*([\s\S]*?)```", raw, re.IGNORECASE)
    if match:
        raw = match.group(1).strip()
    return json.loads(raw)
```

## 当前 Prompt 的已知局限

| 局限 | 影响 | 暂未解决的原因 |
|------|------|--------------|
| 微信公众号文章无法直接提取 | 成功率约 70% | 微信平台限制，非 Prompt 问题 |
| 极短文章（< 200字）提炼质量差 | 要点很少 | 已加 50 字最低长度过滤 |
| 英文文章分类中文化 | topic 会是中文 | 对中文用户来说是正确行为 |
| 同一内容多次保存被去重 | 同 URL 覆盖旧数据 | 设计如此，防止重复累积 |
| 历史数据分类不一致 | 旧数据用自由分类，新数据用受控词表 | 可批量重分类，但需逐步执行 |
