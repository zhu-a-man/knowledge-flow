# AI 提取 Prompt 设计原理

本文记录 `ai_processor.py` 中提取 Prompt 的设计思路，解释「为什么这样写」而不是「写了什么」。

---

## 核心目标：让知识"日后找得回来"

Prompt 设计的出发点不是"提炼准确"，而是**"一个月后的自己能快速找到并看懂"**。

这两个目标的差异很大：
- "提炼准确"→ 倾向于完整、客观、术语准确
- "日后找得回来"→ 倾向于口语化、有钩子感、分类符合检索习惯

---

## 一、为什么角色是"朋友"而不是"知识管理专家"

测试了大量 Prompt 后发现，角色设定对输出风格影响最大。

**"知识管理专家"角色产出**（书面腔，没有阅读欲望）：
```
该工具具备AI辅助代码补全功能，可显著提升开发效率，
适用于专业软件开发场景，支持多种编程语言
```

**"最懂你的朋友"角色产出**（口语化，读完有共鸣）：
```
Cursor — 写代码时 AI 实时帮你补全和改 bug，速度快到不像话
```

**原因**：大模型的风格跟随角色设定。"专家"激活的是学术/报告写作模式，"朋友"激活的是推荐/分享模式。知识库是给自己用的，后者更有效。

---

## 二、为什么用"好/差示例对比"而不是只说规则

单说规则（如"语言通俗易懂"），模型的遵从率低，因为"通俗"本身就是主观的。

加了对比示例后，模型有了锚点：

```
好示例：「Cursor — 写代码时 AI 实时补全，速度快到不像话」
差示例：「Cursor — 一款具备AI辅助功能的代码编辑器」
```

**关键原则**：
- 好示例要体现"帮你解决什么问题"或"为什么值得一试"
- 差示例要是模型最容易产出的那种（功能描述型）
- 每种内容类型（list/insight/tutorial/news）都给一对示例

---

## 三、为什么要明确区分内容类型（list/insight/tutorial/news）

不同类型的内容，提炼的逻辑完全不同：

| 类型 | 提炼重点 | 格式 |
|------|---------|------|
| list（清单） | 原文有几条就提几条，一条不能少 | 名称 — 价值说明 |
| insight（观点） | 3-5 个可复用的结论 | 行动建议或认知翻转句 |
| tutorial（教程） | 关键步骤，"你去做"的语气 | 操作步骤 |
| news（资讯） | 发生了什么 + 为什么关注 | 事件 + 影响 |

没有分类时，模型会用统一格式处理所有内容，导致清单被压缩（原文10条只提5条）、观点被罗列而不是提炼。

---

## 四、为什么清单类要"原文有几条就提几条"

这是版本迭代中发现的最重要规则。

早期 Prompt 没有这条约束，用户反馈"公众号里讲了6个技能，只提取了5个"。

原因：模型默认会"精简"，觉得重复或价值低的条目可以省略。但对用户来说，**清单的完整性比精练更重要**——你不知道哪条会是他下次搜索的关键词。

**正确写法**：
```
【list 清单类】——最重要规则：原文有几条就写几条，一条都不能少
```

注意这里"最重要规则"和"一条都不能少"的强调，会让模型优先遵守。

---

## 五、为什么分类标准是"方便找回"而不是"内容准确"

PKM（个人知识管理）理论中，**知识的组织目的是检索，不是归档**。

两种分类逻辑的区别：
- "归档导向"：这篇文章讲的是 Cursor，所以放在"Cursor"这个 topic 下
- "检索导向"：一个月后我想找这个内容，我会搜什么？ → "AI工具" > "编程助手"

后者更实用，因为你检索时不会记得"Cursor"，但你会记得"上次看到一个 AI 写代码的工具"。

**Prompt 中的表达**：
```
想象用户一个月后想找这篇内容，他会先想到什么词？那就是 topic。
```

---

## 六、为什么不固定 topic 层数

早期版本强制三层分类（topic → dimension → sub-dimension）。

问题：不是所有内容都需要三层。一篇简单的工具推荐文，两层（topic + dimension）足够；一篇深度技术文，可能需要三层。

**改为**：让 AI 自行判断"最合适的粒度"，给出 topic 和 dimension 即可，用 entries 数组支持一篇文章多个条目。这比强制三层更灵活，聚合效果更好。

---

## 七、为什么 summary 要口语化

summary 显示在龙虾的回复里，是用户判断"值不值得点进去看知识库"的第一眼。

**书面腔 summary**：
```
本文介绍了多种提升工作效率的AI工具及其应用场景
```

**口语化 summary**：
```
几个工具装上去效率翻倍，收藏这篇
```

后者更像朋友推荐，有行动感，用户更可能点进去。

---

## 八、JSON 安全解析的必要性

即使 System Prompt 明确写"只返回 JSON，不要任何其他文字"，模型偶尔仍会返回：

````
```json
{"topic": "AI工具", ...}
```
````

直接 `json.loads()` 会抛异常。必须用正则预处理：

```python
def _parse_json_safely(raw: str) -> dict:
    raw = raw.strip()
    match = re.search(r"```(?:json)?\s*([\s\S]*?)```", raw, re.IGNORECASE)
    if match:
        raw = match.group(1).strip()
    return json.loads(raw)
```

## 当前 Prompt 的已知局限

| 局限 | 影响 | 暂未解决的原因 |
|------|------|--------------|
| 微信公众号文章无法直接提取 | 成功率约 70% | 微信平台限制，非 Prompt 问题 |
| 极短文章（< 200字）提炼质量差 | 要点很少 | 已加 50 字最低长度过滤 |
| 英文文章分类中文化 | topic 会是中文 | 对中文用户来说是正确行为 |
| 同一内容多次保存被去重 | 同 URL 覆盖旧数据 | 设计如此，防止重复累积 |
